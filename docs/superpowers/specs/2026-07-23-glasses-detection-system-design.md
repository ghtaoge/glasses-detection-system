# 眼镜识别检测系统设计

## 1. 背景与目标

本项目是一个完全本地运行的计算机视觉教学系统。用户可以准备和标注数据、在 Web 页面启动 YOLO 训练、评估模型，并通过上传图片或浏览器摄像头识别多人画面中的眼镜佩戴情况。

系统固定识别三类人脸：

- `no_glasses`：未戴眼镜
- `eyeglasses`：普通眼镜
- `sunglasses`：墨镜

检测框覆盖完整人脸，不只覆盖镜片。首版模型在冻结的独立测试集上以总体 `mAP@0.5 >= 0.80` 为达标条件，同时必须报告逐类指标和混淆矩阵。该阈值是训练结果的验收门槛，不是预先保证；未达标模型仍可保留和测试，但不会自动成为当前模型。

## 2. 范围

### 2.1 首版包含

- 公开教学数据下载与本地数据导入
- 自动预标注和人工标注工作台
- 不可变数据集版本及训练、验证、测试划分
- Web 训练配置、进度查看、取消、失败诊断和评估
- 模型版本管理、ONNX 导出、测试和手动启用
- 单张图片多人检测
- 浏览器摄像头多人实时检测
- 检测历史、详情、筛选和删除
- CPU 与 NVIDIA CUDA 自动适配
- 后端、前端和端到端测试

### 2.2 首版不包含

- 用户登录、角色权限和多租户
- 云端识别 API
- 分布式训练或任务队列
- 视频文件离线分析
- 镜框颜色、材质、形状和品牌识别
- 移动端原生应用
- 将识别结果用于身份判断、门禁或其他自动决策

## 3. 技术路线

采用单阶段 YOLO 三分类目标检测。模型一次推理输出每张脸的位置、类别和置信度，天然支持多人图片和摄像头画面。相较“人脸检测 + 分类”两阶段方案，该路线具有更统一的标注、评估和部署链路；相较自定义多任务网络，首版实现和教学成本更低。

训练使用固定版本的 Ultralytics YOLO Python 包和小型预训练检测模型。具体包版本与基础权重名称在实现计划中锁定，禁止运行时无版本约束地升级。训练产物导出为 ONNX，在线推理由 ONNX Runtime 执行。参考资料：

- [Ultralytics 训练文档](https://docs.ultralytics.com/modes/train/)
- [Ultralytics 导出文档](https://docs.ultralytics.com/modes/export/)
- [Ultralytics 检测数据格式](https://docs.ultralytics.com/datasets/detect/)

## 4. 总体架构

项目目录为 `glasses-detection-system`，采用本地前后端分离架构。

### 4.1 前端

Vue 3、TypeScript 和 Vite 构建单页工作台，负责：

- 数据集导入和状态展示
- Canvas 标注交互
- 训练参数配置和实时指标展示
- 模型比较、测试和启用
- 图片上传、摄像头采集和检测框绘制
- 检测历史浏览和删除

### 4.2 后端

FastAPI 提供 REST API、训练事件 SSE 和摄像头推理 WebSocket，负责：

- 输入校验和本地文件生命周期
- 数据集、标注和版本管理
- 训练任务调度与状态恢复
- 模型注册、评估、导出和启用
- ONNX 推理和资源协调
- 检测历史与统计查询

### 4.3 持久化

SQLite 保存数据集、图片、标注、训练任务、模型和检测记录等元数据。项目本地受控目录保存原始图片、标注文件、训练产物、ONNX 模型、指标文件及检测结果图。数据库只保存相对于配置存储根目录的路径，所有路径在解析后必须仍位于存储根目录内。

### 4.4 进程模型

API 服务、训练任务和推理运行时相互隔离：

- API 服务不直接执行长时间训练。
- 同一时间只允许一个训练子进程运行。
- 训练子进程通过持久化事件记录上报阶段、进度、日志和指标。
- SSE 只负责传输事件；浏览器断线重连后根据事件序号补取状态。
- 推理服务加载唯一的当前模型，并在模型切换时完成校验后原子替换。

## 5. 模块边界

### 5.1 数据源模块

负责首次联网下载和本地压缩包或文件夹导入。内置教学数据源为 Kaggle 的 [Glasses and Coverings](https://www.kaggle.com/datasets/mantasu/glasses-and-coverings)，仅使用 `plain`、`glasses`、`sunglasses` 和 `sunglasses-imagenet` 目录，其中后两个目录合并为 `sunglasses`。

该数据集标记为 CC BY-NC 4.0，只适用于本项目的非商业教学范围。系统不得把数据打包进仓库；下载前必须展示来源、许可和用途限制并取得用户确认。下载结果记录数据源标识、版本、时间、校验值和许可快照。若 Kaggle 下载方式或许可发生变化，导入必须停止并要求重新确认，不能绕过访问控制。

系统也允许导入用户自己的 JPG、PNG 和 WebP 图片。导入操作复制文件到受控存储，绝不直接修改用户源目录。

### 5.2 预标注模块

公开教学数据是居中的单人分类图片，并非原生检测数据。导入后使用随应用固定版本分发的本地人脸检测模型生成初始人脸框，再把目录类别映射到三类标签。

满足以下任一条件的图片进入“待复核”队列：

- 未检测到人脸
- 检测到多张人脸
- 人脸检测置信度低于配置阈值
- 人脸框面积或宽高比异常
- 图片属于重复或近重复组

自动标注永远是草稿，必须经过数据集发布校验。用户上传的多人图片可包含多个不同类别的人脸框。

### 5.3 标注模块

标注工作台支持绘制、选择、移动、缩放和删除矩形框，支持修改三类标签。坐标以原图像素保存，导出训练数据时统一转换为 YOLO 归一化格式。任何标注变更都记录修改时间；已发布数据集版本不可原地修改，只能从它派生新草稿版本。

### 5.4 数据集版本模块

发布前执行：

- 文件签名、解码、尺寸和像素数校验
- SHA-256 精确去重和感知哈希近重复分组
- 标注非空、类别合法、坐标有效和边界约束校验
- 三类数量、比例和待复核数量检查
- 按重复组分层划分 `70% / 15% / 15%` 的训练、验证和测试集

同一重复组不能跨集合。无法获得人物身份时，系统明确记录“未进行身份级去泄漏”，避免把随机图片划分误述为身份独立评估。每个发布版本生成清单、数据 YAML 和内容摘要，训练任务必须固定引用该版本。

### 5.5 训练模块

Web 页面提供两个预设：

- 快速体验：验证环境和完整链路，不作为模型达标依据。
- 标准训练：用于正式评估，可调整轮次、图像尺寸、批量大小、早停轮次和设备。

后端只接受白名单参数并生成训练命令，不接受任意命令行文本。训练任务状态为 `queued`、`running`、`cancelling`、`completed`、`failed` 或 `interrupted`。取消采用先请求优雅停止、超时后终止子进程的方式，并保留日志和已有检查点。

训练期间持续记录训练损失、验证损失、Precision、Recall、`mAP@0.5` 和 `mAP@0.5:0.95`。完成后在冻结测试集上运行一次独立评估，生成总体和逐类指标、混淆矩阵及错误样本索引。

### 5.6 模型库

模型记录包括来源训练任务、数据集版本、训练参数、代码版本、基础权重、设备、指标、文件校验值和状态。只有满足以下条件的模型可手动启用：

- 训练成功
- ONNX 导出和加载验证成功
- 测试集评估完成
- 类别映射完全匹配三类固定编码

`mAP@0.5 < 0.80` 的模型标记为“未达标”，但允许用户手动测试。达标模型也不会自动替换当前模型。当前启用模型不能删除；必须先启用另一模型或清除当前模型。

### 5.7 推理模块

ONNX Runtime 按 CUDA、CPU 的顺序探测可用执行提供程序。CUDA 不可用时回退 CPU并报告实际设备。推理接口统一返回：

- 图像尺寸
- 每张脸的边界框
- 稳定类别编码和中文显示名称
- 类别置信度
- 推理设备和耗时
- 当前模型标识

图片检测接收单张图片并返回完整结果。摄像头检测通过 WebSocket 接收带单调递增帧序号的压缩图片，只处理当前最新待处理帧，丢弃过期积压帧。响应携带对应帧序号，前端不得把旧响应绘制到新画面。

### 5.8 资源协调模块

训练占用 CUDA 时，新推理会使用 CPU，界面明确显示实际设备。系统不在训练中途静默更换训练设备，因为这会影响可复现性。CUDA 显存不足时训练失败并给出降低批量大小或图像尺寸的建议；用户修改参数后创建新的训练任务。

### 5.9 历史模块

图片检测成功后保存原图、带框结果图、结构化检测项和耗时。摄像头画面只有用户点击“保存快照”时才写入历史，避免逐帧占满磁盘。

历史支持按日期、来源和类别筛选。删除单条或批量记录时，在数据库事务与文件删除之间使用可重试的待删除状态；文件删除失败会保留待清理记录，不返回虚假的成功结果。

## 6. 页面与交互

### 6.1 概览

显示当前数据集版本、三类分布、当前模型、最近训练状态、最近评估结果和检测数量。它是进入各工作流的导航面板，不承担训练或检测操作。

### 6.2 数据集

提供公开数据下载、本地导入、图片列表、异常筛选、类别分布、待复核数量和版本发布。许可确认、下载进度、失败原因和校验状态必须可见。

### 6.3 标注工作台

左侧是可筛选图片队列，中间是稳定尺寸的图片画布，右侧是标注列表和类别选择。画布缩放不改变原图坐标；所有鼠标操作经统一坐标转换层处理。保存后自动进入下一张待复核图片。

### 6.4 训练评估

显示数据版本、预设和可编辑白名单参数。任务运行时展示阶段、设备、耗时、日志尾部和指标曲线，并支持取消。完成后展示逐类表格、混淆矩阵、错误样本和达标状态。

### 6.5 模型库

按版本列出指标、数据版本、设备、创建时间和启用状态。支持比较、样例测试、启用和下载 ONNX。破坏性操作要求二次确认。

### 6.6 识别中心

使用页签切换图片和摄像头模式。

图片模式支持拖放或选择文件，显示原图、检测框、逐人结果、耗时和模型信息。摄像头模式支持设备选择、开始、停止、置信度阈值、目标帧率和保存快照。前端短时间保留最近一次有效框以减少闪烁，超过超时后清空；保留仅影响显示，不会伪造新推理结果。

### 6.7 历史

提供日期、来源和类别筛选，列表展示缩略图、结果摘要、模型和时间。详情页显示原图、结果图和所有检测项。支持单条和批量删除。

## 7. 核心数据流

### 7.1 数据准备

1. 用户确认公开数据许可或选择本地图片。
2. 后端下载或复制图片到暂存区并执行安全校验。
3. 系统计算哈希、去重并运行本地人脸预标注。
4. 用户在标注工作台复核和补充多人场景。
5. 发布服务校验标注、完成分层划分并冻结数据版本。

### 7.2 训练与启用

1. 用户选择数据版本和训练预设。
2. 后端校验参数并创建训练任务。
3. 训练子进程读取固定数据版本并持续写入事件和指标。
4. 训练完成后导出 ONNX，并在冻结测试集上评估。
5. 模型进入模型库，用户查看结果后手动启用。
6. 推理服务校验新模型和类别映射后原子切换。

### 7.3 图片识别

1. 前端上传图片。
2. 后端验证、解码并调用当前 ONNX 模型。
3. 后端保存原图、结构化结果和带框结果图。
4. 前端按原图坐标绘制结果并展示逐人详情。

### 7.4 摄像头识别

1. 浏览器取得用户授权并选择摄像头。
2. 前端按目标帧率压缩当前帧，通过 WebSocket 发送帧序号和图片。
3. 后端丢弃过期帧，对最新帧执行推理并回传相同序号。
4. 前端只绘制不早于当前已展示结果的响应。
5. 用户主动保存快照时，前端发送当前帧和对应检测结果，由后端复核模型与帧信息后写入历史。

## 8. 异常处理与恢复

- 所有 API 错误使用稳定错误码、中文用户说明和服务端关联标识。
- 上传校验真实文件签名、解码结果、文件大小、图像尺寸和总像素数。
- 压缩包导入限制文件数量、单文件大小、总解压大小和压缩比，并拒绝绝对路径、父目录路径和链接文件。
- 下载支持重试和断点续传；最终校验失败时不进入数据集。
- 非法标注不能保存，未完成复核或类别失衡达到阻断阈值时不能发布。
- API 重启后，遗留 `running` 或 `cancelling` 任务转为 `interrupted`，保留日志和检查点供用户创建续训任务。
- SSE 和 WebSocket 断线不会改变任务状态；前端重连后从服务端状态恢复。
- 当前模型缺失、损坏或类别不匹配时推理拒绝启动，不使用随机结果或未声明的备用模型。
- 摄像头拒绝授权、设备被占用或浏览器不支持时，页面提供明确原因并保留图片检测功能。

## 9. 安全与隐私

- 训练和推理不调用云端视觉服务。
- 首次下载只访问配置白名单中的数据和权重来源。
- 后端不接受任意 URL 下载、不执行用户提供的命令、不反序列化不可信 Pickle。
- 训练权重和 ONNX 文件在注册时计算校验值；模型加载限制在受控模型目录。
- 所有本地文件访问经过解析后路径边界检查。
- 原图和人脸数据默认仅保存在项目本地存储目录。
- 用户可以删除检测历史和自有数据集；公开数据源的许可与归属信息随数据版本保留。
- README 必须说明这是教学系统，结果不应用于身份、就业、执法、医疗或其他高风险决策。

## 10. 测试策略

### 10.1 单元测试

- 原图坐标与 YOLO 归一化坐标双向转换
- 标注边界裁剪、空框和非法类别拒绝
- SHA-256 去重、感知哈希分组和跨集合泄漏防护
- 分层划分比例和小样本边界
- 训练任务状态迁移、取消和重启恢复
- 模型达标判定和类别映射校验
- 存储路径边界、压缩包安全和待删除状态
- 摄像头帧序号和过期帧丢弃

### 10.2 后端集成测试

使用临时 SQLite 和临时存储目录覆盖：

- 图片和压缩包导入
- 自动预标注适配器
- 标注保存和数据版本发布
- 模拟训练进程、SSE 重连、取消和失败
- 模型注册、ONNX 冒烟加载和原子启用
- 图片推理、快照保存、历史筛选和删除
- CPU 执行提供程序必测；CUDA 测试按环境条件运行

### 10.3 前端测试

- 标注画布坐标换算和框编辑
- 数据导入、许可确认和发布阻断状态
- 训练事件归并、断线恢复和错误展示
- 模型启用和受保护删除
- 摄像头权限、设备切换、开始停止和旧响应过滤
- 历史筛选、详情和批量删除

### 10.4 端到端测试

浏览器自动化覆盖：

1. 导入小型固定测试数据。
2. 修正自动标注并发布数据版本。
3. 使用模拟训练器完成训练和评估。
4. 启用固定 ONNX 测试模型。
5. 完成多人图片检测并验证检测框。
6. 使用可控视频流模拟摄像头检测和快照。
7. 查看并删除历史记录。

另设一个真实后端 CPU 冒烟测试，验证固定 ONNX 模型可以加载并产生符合接口的数据；持续集成不执行耗时的真实训练。

### 10.5 模型验收

正式训练完成后，在冻结测试集计算总体与逐类 Precision、Recall、`mAP@0.5` 和 `mAP@0.5:0.95`。总体 `mAP@0.5 >= 0.80` 才标记达标。混淆矩阵和错误样本必须可查看，防止总体指标掩盖 `sunglasses` 等少数类问题。

## 11. 运行与兼容性

- 开发目标为 Windows、Python 3.11+ 和现代 Chromium 浏览器。
- CPU 是必需的基线运行路径；NVIDIA CUDA 是可选加速路径。
- 首次安装允许联网下载固定版本依赖、预训练权重和经用户确认的数据集。
- 首次资源准备完成后，数据标注、训练、评估、推理和历史管理均可离线运行。
- 前后端分别提供开发启动命令，并提供统一的本地启动脚本；容器化不属于首版必需验收项。

## 12. 完成标准

满足以下条件时首版完成：

- 七个工作台页面和约定交互可用。
- 无现成数据时可通过许可确认、下载、预标注和人工复核形成首个数据版本。
- Web 页面可启动、观察、取消并恢复训练工作流。
- 训练产物可导出为 ONNX、评估、注册并手动启用。
